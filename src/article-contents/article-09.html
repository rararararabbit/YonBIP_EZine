编者按：你有没有想过，在和客户做初次交流的时候，演示的都是客户自己的数据，介绍的都是客户熟悉的业务，此时你是怎样的一种感受？客户又是怎样的一种体验？在企业知识治理场景，Skill可以帮你做到这一切。


一、引言：从售前困境说起

在售前客户交流环节，如果我们只是拿着早已备好的千篇一律的案例来演示，现在已很难打动客户。客户知道那是一份久经考验并且和他们关系不大的数据，基本不会出什么问题。真正能引起共鸣、触动人心的，往往是用客户最熟悉的数据和业务，来完成一次企业 AI 的交流演示。当客户看到自己的官网、自己的产品目录、自己的业务板块出现在实操中时，那种信任感是“预制菜”类型的 Demo 所无法比拟的。


在企业知识治理场景，我们已经可以实现这种“量身定制”的效果。这在很大程度上要归功于Skill 的创新应用——将系统性获取和利用客户公开知识的能力封装为可复用的自动化工具，让每一次客户交流都能快速、真实地获取客户上下文，做到千人千面。


本文以 web-knowledge-harvest（网站知识获取技能）为例，展示如何用 Skill + AI 的方式实现这一模式。企业知识治理还有很多体系化工作，使其 AI 化、技能化将是我们后续分享的方向。

二、Skill + AI：让“走近客户”从理念走向实践

“走近客户、了解客户”是每个售前和研发人员的共识，但知易行难。传统的做法是人工浏览客户官网、下载资料、整理成文档——不仅效率低，而且难以持续，换一个客户就要重新来一遍，同一客户的资料也会经常翻新。技能的引入为解决这一难题提供了新的思路：Skill 将特定的业务或技术能力封装为可复用、可组合的自动化工作流。从知识获取、规范检查到内容生成，Skill 生态已在逐步覆盖研发全场景。


以 web-knowledge-harvest 为例：它实现的是“将客户官网的公开知识转化为本地知识库”这一能力。类似的 Skill 还可以组合使用——获取知识后，可以进一步调用进行内容摘要、标签提取、自动分类等操作。这恰恰是 AI 可以充分发挥作用的领域：Skill 负责自动化采集和整理，AI 负责理解和推理，二者结合，才能真正实现“用客户的数据打动客户”。


三、web-knowledge-harvest 的设计思路

3.1 核心能力

web-knowledge-harvest 的核心能力是：自动解析目标网站的主菜单结构，识别菜单层级，然后按目录下载文档（带图文）、图片和视频到本地。文件的命名规则以菜单层级为前缀（一级-二级-三级-标题-统一后缀.docx），这不仅避免了重名，更隐含了分类体系的层级关系——为后续的知识治理埋下了结构化的伏笔。


3.2 关键设计决策

Ø 全站模式 vs 专题模式。通过 --section 参数切换：不加参数时，技能解析全站菜单，遍历所有栏目下载；加上 --section 后，跳过菜单解析，直接处理指定 URL 的内容及分页。这一设计覆盖了“首次全量获取”和“局部补充更新”两类核心场景。

Ø 菜单预览先行。--menu-only 参数让用户可以先看网站菜单结构树，再决定实际下载范围，做到心中有数再动手。

Ø 效率与投入产出比。技能通过多重机制避免过量下载：文档列表页上限 ** 篇、图片上限 ** 张、视频上限 ** 个（均可配置）；单文件超过阈值自动跳过；过小文件视为装饰性内容自动过滤；请求间加入随机延迟防止对目标网站造成压力。不求大而全，重点获取典型代表性知识，确保每一次运行的投入产出比。


3.3 目录结构规范

下载后的知识库按网站菜单层级组织目录结构：

知识库根目录/

├── 一级菜单/

│ ├── 二级菜单/

│ │ ├── 三级菜单/

│ │ │ └── 一级-二级-三级-标题.docx

│ │ └── 一级-二级-标题.docx

│ └── 一级-标题.docx

├── 知识图片/

└── 知识视频/


这个目录结构看似简单，实则是一种轻量级本体——菜单层级约等于分类体系，文件名前缀约等于实体标签，文档内容约等于实体属性。当后续需要构建知识图谱时，这种结构化布局可以直接映射为“概念-关系-实例”的三元组骨架，避免从零开始的洗数成本。


3.4 参数设计：一套入参覆盖多场景

技能提供了多个可选参数，通过参数组合即可适配不同场景，如 –clean实现文件名净化、--section 实现局部补充、--menu-only 实现菜单预览、--include 实现栏目过滤、--force 实现增量重下等。参数之间可以自由组合。这种设计理念是：一个工具覆盖从预览、采集、补充到治理的全流程，避免为不同场景开发多个工具。


四、实战验证：在差异化挑战中打磨方案

不同网站的导航结构千差万别，菜单解析是系统化知识获取的核心难点，也是模拟企业知识库建设的关键环节。本文验证中，我们选取了五家企业的官网对该模式进行了验证：攀钢集团（传统门户，iframe 嵌套导航）、鞍钢集团（标准门户，PDF 多）、福建能化集团（.NET 架构，POST 表单翻页）、爱尔眼科（多级菜单，概述页与列表页混杂）、用友集团（产品方案类，多分类体系）。


Ø 爬坑进化史。不同的企业官网，从组件化框架、UI 样式框架到轻量化方案、静态方案，各有各的特色。最开始解析传统面包屑导航的布局，等换到动态渲染的布局，信息识别就必须变得更聪明；前面解决了由 GET 实现的表单翻页，后面又会遇到由 POST 实现的表单分页；有的非末级菜单页是个空壳、有的非末级菜单页又变成了详情页；同一套下载文件命名策略，换个网站就一片大乱。正是这样不断遇坑、不断进化，方案的能力和兼容性才一步步增强。

Ø 有益的启示。每家企业的网站都是其组织架构和信息治理水平的一面镜子。解析菜单结构的过程，本身就是一次对客户信息体系的“逆向建模”。而这种“逆向建模”的产物——菜单树加文档内容——天然构成了领域本体的雏形。

Ø 特别说明。对于完全由 JS 动态渲染的网站（如 Vue/React SPA），其菜单结构和正文内容在 HTML 源码中不可见，分析难度大幅提升。技能已内置 SPA 检测与 API 接口回退机制，可应对部分场景。随着技能不断经受各类网站的训练，日臻成熟，相信会积累出更好的解决逻辑。


五、从“获取”到“治理”——构建客户基础知识库

有了前面的基础，我们来看如何将一次性的采集转化为可持续的知识治理，从而支撑售前场景的量身定制。

1. 系统性采集。通过 Skill 一键执行，标准化的目录结构自动生成。以爱尔眼科为例，运行后 8 个一级栏目、数十个子栏目的内容被自动归类到对应目录中，文件名一目了然。这相当于快速搭建了一个客户专属的迷你知识库底座。

2. 分类整理。按官网菜单层级自然形成分类体系，无需人工干预。这份分类体系本身就是对客户业务架构的一次梳理。

3. 知识呈现。有了这个底座，售前人员在客户交流前，可以快速浏览客户的最新动态、业务布局、产品体系；在 POC 环节，可以用客户自己的产品数据、新闻案例来做演示，让客户感受到“这就是为我准备的”。


这类基础知识库虽然朴素，但它已经具备了结构化的骨架。没有这个基础，上层 AI 能力就是空中楼阁。通过基础知识库构建，售前人员可快速掌握客户情况，准备更有针对性的方案；研发人员理解客户行业术语和业务重点，产品设计更贴合场景。更重要的是——通过 Skill 获取客户的背景资料和运营情况，不只让我们手头有了素材，更在过程中增进了对客户及其业务的了解。当我们比客户自己还熟悉他们的官网时，距离自然就近了。


六、从“走近客户”到“赋能客户”

6.1 从“千篇一律”到“量身定制”——Skill 思维的价值

用友服务数百万企业客户，自身的研发治理水平是客户信任的基石。无论是 web-knowledge-harvest 还是其他 Skill，它们共同体现了用友研发团队系统化、工程化、可复用地解决问题的治理思维。这种思维映射到售前场景，就是用客户自己的数据来呈现方案价值——让每一次交流都成为一次“量身定制”的对话。

6.2 走近客户才能更好地服务客户

当我们系统性获取一家客户的官网知识时，实际上是在模拟客户自己的知识治理过程。客户官网上的信息散乱、更新滞后、结构不清——这恰恰是许多企业面临的知识治理困境。用友通过自身的 Skill + AI 实践，不仅学会了解客户，更积累了帮助客户进行知识治理和数智化转型的方法论。


6.3 从“基础知识库”管窥“企业AI蓝图”

知识治理是一个不断迭代的过程，本文展示的“系统性获取、分类整理、本地化管理”是企业知识治理的基石。这个基石之上，可以构建本体建模、知识图谱、智能问答、行业知识沉淀、精准营销支撑等一系列高级能力。需要说明的是，基于 Skill 获取客户公开知识，只是整个企业知识治理演示过程的第一步。后续还有知识库建设、标注、索引、切片、本体化等一系列工程——而在这些环节中，AI 与 Skill 都将发挥巨大的作用。


6.4 用友的思路：先治理自身，再赋能客户

我们在用友内部做知识治理的每一次探索——从 Skill 获取客户信息，到分类整理，再到结构化存储——都是在积累可复用的方法论。这些方法论最终会沉淀到 BIP 平台的产品能力中，让我们的客户也能用同样的方式治理自己的企业知识。这正是用友“企业数智化专家”定位的技术落地——先做好自己的治理，才能赋能客户的治理。
