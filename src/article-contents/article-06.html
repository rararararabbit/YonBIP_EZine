Agent Skills 不是 Prompt，也不是 MCP：一文讲清 AI 智能体的“工作手册”

最近很多人在讨论 Agent Skills。它看起来像 Prompt，又经常和 MCP、Function Calling 一起出现，于是很容易被混成一个概念：是不是把一段提示词包装一下，就叫 Skill？是不是 MCP 工具越多，Skill 就越强？

都不是。

如果只记一句话：Prompt 说的是“这次要做什么”，MCP 和 Function Calling 解决“能调用什么”，Agent Skills 沉淀的是“一类任务应该怎么做”。

这篇文章把它拆开讲清楚：Skill 到底是什么，它和 Prompt、MCP、Function Calling 分别处在什么位置，SKILL.md 应该怎么写，以及真正落地时最容易踩的坑。

先给结论：Skill 是给 Agent 看的工作手册

Agent Skills 不是神秘的新模型能力，也不是某个工具函数。

它更像一份可被 Agent 发现、按需加载的任务说明书。里面可以写流程、规则、边界、默认方案、失败处理、脚本入口和参考资料。

比如团队里有一套代码审查规范：

- 先看架构和模块边界，再看实现细节；
- 涉及数据库查询时检查软删除条件；
- 涉及用户输入时检查越权、注入和 XSS；
- 输出问题时按严重程度排序，不要只挑格式。

这些内容当然可以每次粘进 Prompt，但问题也明显：换个会话要重粘，换个同事还要重粘，规范一长，模型还可能漏掉中间几段。

把它沉淀成 Skill 后，Agent 在遇到“代码审查”这类任务时，再加载对应的 SKILL.md，并按里面的流程执行。它不是替代人的判断，而是把重复提醒变成稳定的工作流。

Prompt、Function Calling、MCP、Skill 到底差在哪

先把四个概念放到同一条执行链路里看。

用户说：“帮我分析这份报表。”这句话是 Prompt，表达的是本次任务目标。

模型判断需要读取文件，于是生成一个结构化工具调用，这一步是 Function Calling。

如果读取文件、查数据库、调 GitHub 这些能力来自 MCP Server，那 MCP 负责的是工具连接、协议和能力暴露。

但“分析报表时先理解字段，再看异常值，最后给业务结论，不要直接堆统计指标”这种经验，不属于 Prompt，也不属于 MCP，更不属于 Function Calling。它适合放进 Skill。

可以这样区分：

概念
解决的问题
典型内容
Prompt
用户这次要做什么
“帮我审查这段代码”“分析这份报表”
Function Calling
模型如何发起工具调用
函数名、参数、结构化调用
MCP
外部能力如何接进来
文件系统、数据库、浏览器、GitHub、公司内部服务
Skill
这类任务应该怎么做
流程、约束、默认方案、检查清单、失败处理

所以，Skill 不是 MCP 的竞品。MCP 更偏“连接能力”，Skill 更偏“执行方法”。一个复杂 Agent 往往会同时用到它们：Skill 指挥流程，Function Calling 发起动作，MCP 提供工具通道。

一个 Skill 通常长什么样

最小可用的 Skill 可以只有一个目录和一个 SKILL.md：

code-review-skill/
├── SKILL.md
├── scripts/
├── references/
└── assets/

SKILL.md 通常分两块。

第一块是 YAML frontmatter，也就是元数据：

第二块是正文，写真正的执行说明：

Code Review Skill
## Workflow
1. 先获取变更文件列表和 diff。
2. 第一轮检查设计和模块边界。
3. 第二轮检查正确性、安全、性能和边界条件。
4. 只输出会影响行为、稳定性或维护成本的问题。
## Output
按严重程度排序，给出文件、行号、原因和建议修改方式。

元数据负责“能不能被发现”，正文负责“被发现后怎么执行”。

description 比很多人想得更重要

Agent 一般不会一开始就把所有 Skill 全部读进上下文。它通常先看到每个 Skill 的 name 和 description，再判断当前任务该不该加载完整 SKILL.md。

这意味着 description 不是简介文案，而是路由规则。

写 description 时，建议至少说清楚两件事：

1. 这个 Skill 做什么；
2. 用户在什么场景下会触发它。

不太好的写法：

description: 帮助处理文档

更好的写法：

description: 从 PDF、Word、Markdown 中提取正文、表格和图片，并生成可发布的结构化文档。在用户提到解析文档、转换格式、提取表格、生成报告时使用。

后者不是更“华丽”，而是更容易被路由命中。它同时包含能力、场景和用户可能说出口的触发词。

正文别写成 README，要写成操作手册

很多 Skill 写不好，是因为写作者把它当成项目 README。

README 是给人看的，可以解释背景、理念、安装方式和上下游资料。Skill 是给 Agent 执行任务看的，最值钱的是那些 Agent 猜不到、但做错就会出问题的约定。

比如下面这句很有价值：

users 表使用软删除。所有正式查询都必须加 WHERE deleted_at IS NULL。

下面这种就不太值得放进 Skill：

软删除是一种常见的数据删除方式，通常不会真正删除数据库记录。

前者是项目私有规则，Agent 无法凭常识稳定猜到。后者是通用解释，会占上下文，却不能提升执行质量。

写正文时可以反复问三个问题：

- Agent 执行任务时真的需要这段吗？
- 这是项目私有知识，还是通用科普？
- 这段能不能变成明确动作、默认方案或失败处理？

延迟加载：为什么不要把所有资料一次性塞进去

Skill 的一个核心价值，是让上下文按需增长。

如果把所有规范、教程、API 文档、案例都一次性塞给 Agent，看起来很完整，实际会制造噪声。上下文越长，关键信息越容易被稀释；一些重要限制条件夹在中间，还可能被模型忽略。

更稳的方式是渐进式披露：

1. 启动时只暴露 name 和 description，让 Agent 知道有哪些 Skill；
2. 命中任务后加载 SKILL.md 主流程；
3. 执行到特定分支时，再读取 references、scripts、assets。

比如数据分析 Skill 可以这样组织：

bigquery-analysis/
├── SKILL.md
└── references/
    ├── finance.md
    ├── sales.md
    ├── product.md
    └── marketing.md

用户问“上个季度销售管道怎么样”，Agent 只需要打开 sales.md。财务、产品、营销资料不需要进上下文。

主文件应该像目录和调度器，而不是资料仓库。它要告诉 Agent：现在有哪些路，什么情况走哪条路。

自由度怎么给：高风险任务要窄，判断型任务要留空间

Skill 不是越详细越好，也不是越自由越好，关键看任务出错的代价。

数据库迁移、生产部署、批量删除、权限变更这类任务，应该低自由度。命令、参数、顺序、验证点、失败后是否停止，都要写清楚。

例如：

数据库迁移：
1. 运行 python scripts/migrate.py --backup --verify
2. 不要修改命令，不要添加额外参数
3. 如果命令失败，停止执行，并把错误输出返回给用户

代码审查、方案评估、文章改写这类任务，则需要中高自由度。你可以固定检查维度和输出优先级，但不要把每一步都写死，否则 Agent 反而难以结合上下文判断。

一个实用判断：

任务类型
自由度
Skill 应该写什么
部署、迁移、删数据
低
精确命令、停止条件、回滚和验证
格式转换、报表生成
中
默认工具、模板、异常分支
代码审查、技术评估

检查维度、输出标准、优先级


复杂任务一定要写验证点

很多失败不是因为 Agent 不懂概念，而是因为流程里少了“确认没跑偏”的环节。

以 TDD 为例，只写“先写测试再写代码”并不够。更关键的是 Verify RED：必须先看到测试失败，并确认失败原因是功能缺失，而不是路径写错、语法错误或测试本身崩了。

一个更可靠的 Skill 会把反馈循环写出来：

运行 -> 验证 -> 修复 -> 再验证

代码审查也一样。不要只写“全面审查代码”，可以拆成两轮：

1. 第一轮看架构、模块边界、核心数据流；
2. 第二轮看正确性、安全、性能、异常处理；
3. 输出时先列会影响线上行为的问题，再列维护性建议。

检查清单要写成可判断的动作，比如“所有新增函数都有测试”“本地图片路径都存在”“发布命令返回 mediaId”。不要写“保证质量”“遵循最佳实践”这种无法验收的大话。

Skill 路由：别让 Agent 在一堆说明书里迷路

Skill 少的时候，靠 description 判断就够了。Skill 多起来以后，路由会变成一个小型检索问题。

一个轻量可行的方案是：

1. 粗召回：把 name、description、典型 query 样本向量化，取 top-5；
2. 精排：同时命中标题、描述和触发词的优先级更高；
3. 兜底：最高分很低时，不选 Skill，走默认流程。

这里最怕的不是“不够智能”，而是选错。选错 Skill 后，后续工作流可能整条跑偏。

新 Skill 还有冷启动问题：没有历史 query，description 又写得太泛，检索就容易漂。可以在元数据里增加 triggers 字段，把用户常见说法写进去：

name: jvm-runtime-diagnosis
description: Diagnose Spring Boot production runtime issues including OOM, Full GC, high CPU, slow APIs, and thread deadlocks.
triggers:
  - 接口卡死了
  - 频繁 Full GC
  - 服务 OOM 了怎么排查
  - 帮我看看这段 Java 堆栈

这不是为了好看，而是给路由器补样本。

写 Skill 最容易踩的 8 个坑

最后把坑集中列一下，真正落地时很常见。

第一，把 Skill 当 README 写。背景很多，动作很少，Agent 读完仍然不知道下一步做什么。

第二，一个 Skill 想包打天下。代码审查、数据库排查、部署、文档生成全塞一起，边界不清，路由也会飘。

第三，description 太虚。只写“帮助提升效率”，没有任务场景，也没有触发词。

第四，给 Agent 太多同级选择。比如 PDF 处理同时列 pypdf、pdfplumber、PyMuPDF、OCR，却不写默认方案和例外条件。

第五，术语来回换。同一个概念一会儿叫 endpoint，一会儿叫 URL，一会儿叫 route，条件判断会变得不稳定。

第六，让 LLM 做确定性工作。批量转换、精确计算、格式校验、文件改名这类事，能交给脚本就交给脚本。

第七，没有失败处理。只写成功路径，不写命令失败、权限不足、配置缺失、网络超时时应该停还是重试。

第八，没有验证清单。Agent 做完就说完成，但没有检查输出文件、测试结果、图片路径、发布返回值。

最后再收一下

Prompt、Function Calling、MCP、Skills 不是同一层的四个竞品。

Prompt 是这次任务；Function Calling 是模型发起工具调用的方式；MCP 是外部能力接入协议；Skill 是一类任务的工作手册。

写好 Skill 的核心，不是把文档写长，而是把任务边界、默认路径、关键约束、失败处理和验证点写清楚。

真正好用的 Skill，应该让 Agent 少猜一点、少漏一步、少把确定性工作交给语言模型硬想。

当团队开始把这些经验沉淀下来，Agent 才不只是“这次回答得不错”，而是能在同一类任务上稳定复现更可靠的工作方式。
